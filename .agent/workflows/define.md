## ⚠️ 交付标准（写任何内容之前必须先读这一节）

你撰写的 spec.md 最终必须同时满足以下 6 项硬性指标，**任何一项不通过都禁止提交**：

| # | 自检项 | 合格标准 |
|---|--------|----------|
| 1 | `file:///` 可点击源码链接 | 文档中 ≥ 3 处。格式：`[文件名:L行号](file:///绝对路径#L行号)`。从 `grep_search`/`view_file` 返回值中复制绝对路径，反斜杠换正斜杠。示例：`[index.html:L78](file:///c:/Users/Firekula/.CocosCreator/packages/mcp-inspector-bridge/src/panel/index.html#L78)`。**纯文字行号（如"第 78 行"）不合格。** |
| 2 | ASCII 布局图 | 视觉需求章节必须含 ≥ 1 张 ASCII 字符布局图，画出修复前后的元素排列对比。禁止仅用文字描述。 |
| 3 | 代码片段 | ≥ 2 处 fenced code block（根因原始代码 + 修复方案代码） |
| 4 | 边界情况 | ≥ 4 条，每条含：场景 + 风险 + 缓解策略 |
| 5 | 现有机制复用说明 | 明确列出复用了哪些已有变量/函数/CSS 类，以及为什么不需新增 JS 逻辑（如适用） |
| 6 | 文件清单表格 | Markdown 表格格式，列出所有改动文件 |

## 执行指令

### 1. 上下文加载
* 阅读 `memory/constitution.md` 确认技术栈与编码规范约束。
* 仔细查看用户附带的所有图片（截图/设计稿）。

### 2. 深度代码调研（禁止跳过）

**定量红线**：调研阶段必须执行 `grep_search` 或 `view_file` **至少 5 次**才允许进入撰写阶段。不足 5 次说明调研深度不够。

必须完成以下调研动作：
* **定位受影响代码**：用 `grep_search` 搜索至少 2 个关键词（如类名、CSS 选择器、函数名），用 `view_file` 读取每个命中文件的**完整上下文**（前后至少 30 行）。
* **追溯现有机制**：找到相关的 state 变量、computed 属性、composable 函数，理解它们的读写链路。必须回答："现有的哪些机制可以复用？哪些不能？"
* **历史修复记录**：搜索 `UPDATE_LOG.md` 和 `README.md`，确认是否有过类似改动或回归风险。
* **严禁**仅凭用户描述就开始撰写规范，必须用代码事实支撑每一个结论。
* **记录绝对路径**：调研过程中，将每个相关文件的绝对路径（从工具返回值中获取）记录下来，后续撰写时直接用于构建 `file:///` 链接。

### 3. 起草规范 (Spec)
* 仅创建/更新**一个文件**: `specs/[feature_name]/spec.md`。
* **禁止**创建 checklist 清单文件。
* **禁止**创建单独的分析文件。
* **语言要求**: 所有输出及文档内容必须使用**简体中文 (Simplified Chinese)**撰写。

### 4. 文档结构与格式要求

#### 4.1 背景
* 简述问题/需求的来龙去脉。
* **每个技术结论必须附带可点击的源码链接**（格式见上方交付标准表第 1 项）。
* 如有相关历史改动，引用 `UPDATE_LOG.md` 中的版本号和条目标题。

#### 4.2 视觉需求 (Visual Requirements)
* **必须包含 ASCII 布局图**，用代码围栏包裹，画出改动前后的元素排列对比。参考示例：
  ```
  正常宽度:  [按钮组]  [分辨率选择框 ▼]  [旋转]  ···  品牌文字
  极窄宽度:  [按钮组]  [旋转]           品牌文字
             [分辨率选择框（独占整行）▼            ]
  ```
* 禁止仅用文字描述视觉效果。

#### 4.3 功能需求 (Functional Requirements)
* **根因分析**：逐条列出每个技术原因，每条必须附 `file:///` 源码链接。
* **具体修复方案**：按改动点逐个列出，每个点必须包含：
  - 目标文件与行号（`file:///` 链接格式）
  - **改动前后的代码片段**（用 fenced code block）
  - 如涉及 CSS `order`、`z-index`、`flex` 等布局属性的交互，必须解释元素间的排列关系
* **现有机制复用清单**：用列表明确列出"复用了哪些已有变量/函数"及理由。

#### 4.4 涉及文件清单
* Markdown 表格列出所有需改动文件、改动类型（新增/修改/删除）和说明。

#### 4.5 边界情况 (Edge Cases)
* **至少列出 4 条**边界场景。
* 每条必须包含：场景描述 + 风险分析 + 缓解策略。
* 典型必查边界：临界值抖动、溢出/越界、与其他响应式机制的冲突、高度变化对下游布局的影响。

### 5. 出厂自检（强制执行）

在创建 spec.md 之后，**必须立即用 `grep_search` 对自己写的 spec.md 文件搜索 `file:///`**：
- 如果搜索结果数 < 3，**必须立即编辑 spec.md 补充链接直到 ≥ 3**，然后再次搜索验证。
- 同时检查边界情况条目数是否 ≥ 4，不足则补充。
