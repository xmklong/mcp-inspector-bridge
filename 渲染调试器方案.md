# Cocos Creator 2.4.x 运行时渲染调试器 完整实现方案
本方案**100%适配Cocos Creator 2.4.0~2.4.13全版本**，对标Unity Frame Debugger核心能力，无侵入式实现（不修改引擎源码），兼容WebGL/原生iOS/Android双平台，核心解决2.4.x 2D渲染的DrawCall拆解、合批失败定位、渲染流程回放、渲染异常排查四大核心痛点。

---

## 一、前置核心基础：Cocos 2.4.x 渲染底层原理
本方案完全基于2.4.x原生渲染架构设计，先明确引擎渲染全流程与合批规则，是实现的核心前提（所有钩子、分析逻辑均基于此）。

### 1.1 单帧渲染全流程（可拦截节点标注）
```
导演mainLoop → drawScene场景遍历
  → 渲染组件(Sprite/Label等)执行render
  → 【钩子1】pushRenderCommand 提交RenderCommand到渲染队列
  → 所有节点渲染完成，执行cc.renderer.render
    → 渲染队列按globalZOrder排序
    → 【钩子2】遍历命令执行execute，进入合批逻辑
      → 【钩子3】cc.renderer._batcher.addQuad 提交Quad到合批器
      → 合批条件不满足时，触发【钩子4】_batcher.flush 提交DrawCall
    → 【钩子5】cc.gfx.Device.draw 最终提交GPU绘制
→ 帧缓冲交换，画面呈现
```

### 1.2 2.4.x 自动合批**严格必要条件**（100%基于引擎源码）
合批失败是90%的2D渲染性能问题根源，本方案的合批分析完全基于此规则：
1.  命令必须是`QuadCommand`（Sprite、BMFont Label等），`CustomCommand`/`Graphics`/`Mask`会强制打断合批
2.  必须使用**同一个材质实例**（引用完全一致，相同内容的不同实例也会打断合批）
3.  绑定的主纹理`Texture2D`必须是同一个对象引用
4.  材质的`blendFunc`混合模式完全一致
5.  合批过程中材质的Uniform、深度/模板测试等渲染状态无修改（修改会强制flush）
6.  命令的`globalZOrder`完全一致，排序后连续
7.  总顶点数不超过引擎上限（默认4096个顶点，即1024个Quad）

## 二、方案核心目标与整体架构
### 2.1 核心目标
1.  逐帧全量捕获渲染命令流与真实GPU DrawCall，完整还原渲染过程
2.  支持渲染暂停、单步DrawCall回放，可视化查看每一步的绘制结果
3.  精准定位合批失败根因，给出可落地的优化建议
4.  全参数拆解DrawCall：纹理、材质、Shader、混合模式、顶点数据、来源节点
5.  低侵入、低性能损耗，运行时可用，正式包可一键剔除
6.  兼容WebGL与原生JSB平台

### 2.2 整体架构（低耦合分层设计，可直接集成到运行时调试插件）
| 层级 | 核心职责 |

## 三、分模块详细实现方案
### 3.1 渲染钩子层（核心基石，无侵入式实现）
通过JS原型注入实现，调试器开启时注入钩子，关闭时100%恢复引擎原生方法，零侵入引擎源码。

#### 3.1.1 核心拦截点与注入逻辑
##### （1）帧生命周期钩子：标记帧边界，控制渲染暂停
拦截`cc.Director.prototype.mainLoop`与`cc.renderer.render`，标记每帧的开始/结束，实现渲染暂停控制。
```typescript
// 保存原生方法
private _originDirectorMainLoop: Function;
private _originRendererRender: Function;

// 注入帧生命周期钩子
private _injectFrameHook() {
    this._originDirectorMainLoop = cc.Director.prototype.mainLoop;
    cc.Director.prototype.mainLoop = (dt: number) => {
        // 暂停状态：仅执行调试器UI更新，跳过游戏场景渲染
        if (this.isPaused) {
            this._updateDebugUI(dt);
            return;
        }
        // 帧开始：清空临时捕获数据
        this._resetFrameTempData();
        this._currentFrameId++;
        // 执行原生主循环
        this._originDirectorMainLoop.call(cc.director, dt);
        // 帧结束：生成帧快照
        this._generateFrameSnapshot();
    };

    // 注入渲染结束钩子，标记帧渲染完成
    this._originRendererRender = cc.renderer.render;
    cc.renderer.render = (scene: cc.Scene) => {
        this._isRendering = true;
        this._originRendererRender.call(cc.renderer, scene);
        this._isRendering = false;
    };
}
```

##### （2）渲染命令提交钩子：捕获所有渲染元素的来源
拦截`cc.renderer.pushRenderCommand`，捕获所有提交的RenderCommand，关联来源节点与组件，排除调试器自身的渲染命令。
```typescript
private _originPushRenderCommand: Function;
private _injectCommandHook() {
    this._originPushRenderCommand = cc.renderer.pushRenderCommand;
    cc.renderer.pushRenderCommand = (command: any) => {
        // 跳过调试器自身的渲染命令
        if (command.node && command.node.group === 'debugger') {
            this._originPushRenderCommand.call(cc.renderer, command);
            return;
        }
        // 结构化存储命令数据
        const commandInfo: IRenderCommandInfo = {
            commandId: this._commandIdCounter++,
            type: command.constructor.name,
            node: command.node,
            component: command.component,
            zOrder: command.zOrder,
            globalZOrder: command.globalZOrder,
            material: command.material,
            texture: command.texture,
            blendFunc: command.blendFunc,
            quadCount: command.quadCount || 0,
            isBatched: false,
            belongDrawCallId: -1,
            vertexData: command.vertices,
            indexData: command.indices
        };
        this._currentFrameTempData.commands.push(commandInfo);
        // 执行原生提交逻辑
        this._originPushRenderCommand.call(cc.renderer, command);
    };
}
```

##### （3）合批核心钩子：捕获合批过程与中断点
拦截`cc.renderer._batcher.addQuad`与`cc.renderer._batcher.flush`，这是2.4.x合批的核心入口，精准捕获合批中断的时机与原因。
```typescript
private _originBatcherAddQuad: Function;
private _originBatcherFlush: Function;
private _injectBatchHook() {
    const batcher = cc.renderer._batcher;
    this._originBatcherAddQuad = batcher.addQuad;
    this._originBatcherFlush = batcher.flush;

    // 拦截Quad提交，捕获合批状态
    batcher.addQuad = (...args: any[]) => {
        const [material, texture, blendSrc, blendDst] = args;
        // 对比当前合批状态，预判是否会触发flush
        const willBreak = this._checkBatchWillBreak(material, texture, blendSrc, blendDst);
        if (willBreak) {
            this._currentFrameTempData.pendingBreakInfo = {
                breakMaterial: material,
                breakTexture: texture,
                breakBlend: { src: blendSrc, dst: blendDst },
                breakCommand: this._currentFrameTempData.commands.at(-1)
            };
        }
        // 执行原生addQuad
        this._originBatcherAddQuad.call(batcher, ...args);
    };

    // 拦截flush，捕获真实DrawCall触发
    batcher.flush = () => {
        // 仅捕获游戏渲染的flush，跳过调试器自身的
        if (this._isRendering && batcher._verticesBuffer.byteOffset > 0) {
            this._captureDrawCall(batcher);
        }
        // 执行原生flush
        this._originBatcherFlush.call(batcher);
    };
}

// 预判合批是否会中断
private _checkBatchWillBreak(material: cc.Material, texture: cc.Texture2D, blendSrc: number, blendDst: number): boolean {
    const batcher = cc.renderer._batcher;
    // 完全匹配引擎原生合批判断逻辑
    return batcher._material !== material ||
        batcher._texture !== texture ||
        batcher._blendSrc !== blendSrc ||
        batcher._blendDst !== blendDst;
}
```

##### （4）最终DrawCall钩子：捕获GPU真实绘制调用
拦截`cc.gfx.Device.prototype.draw`，捕获所有平台最终提交到GPU的DrawCall，兜底校验DrawCall数量与参数。
```typescript
private _originDeviceDraw: Function;
private _injectDrawCallHook() {
    this._originDeviceDraw = cc.gfx.Device.prototype.draw;
    cc.gfx.Device.prototype.draw = (...args: any[]) => {
        const [primitiveType, indicesStart, indicesCount] = args;
        // 记录DrawCall基础信息
        this._currentFrameTempData.finalDrawCalls.push({
            primitiveType,
            indicesStart,
            indicesCount,
            timestamp: performance.now()
        });
        // 执行原生绘制
        this._originDeviceDraw.call(cc.gfx.Device.getInstance(), ...args);
    };
}
```

##### （5）钩子恢复逻辑：关闭调试器时100%还原引擎
```typescript
public dispose() {
    // 恢复所有原生方法
    cc.Director.prototype.mainLoop = this._originDirectorMainLoop;
    cc.renderer.render = this._originRendererRender;
    cc.renderer.pushRenderCommand = this._originPushRenderCommand;
    const batcher = cc.renderer._batcher;
    batcher.addQuad = this._originBatcherAddQuad;
    batcher.flush = this._originBatcherFlush;
    cc.gfx.Device.prototype.draw = this._originDeviceDraw;
    // 释放所有快照数据
    this._clearAllSnapshot();
}
```

### 3.2 帧数据捕获层：结构化帧快照设计
捕获钩子数据后，生成可回放、可分析的结构化帧快照，同时管理内存，避免溢出。

#### 3.2.1 核心数据结构定义
```typescript
// 帧快照：单帧完整渲染数据
export interface FrameSnapshot {
    frameId: number;
    totalDrawCalls: number;
    totalQuads: number;
    totalVertices: number;
    totalIndices: number;
    renderCommands: IRenderCommandInfo[]; // 所有提交的渲染命令
    drawCalls: IDrawCallInfo[]; // 最终GPU DrawCall列表
    batchBreaks: IBatchBreakInfo[]; // 合批中断点列表
    frameBuffer: HTMLCanvasElement | Uint8Array; // 本帧完整渲染结果
    timestamp: number;
}

// 渲染命令信息
export interface IRenderCommandInfo {
    commandId: number;
    type: string;
    node: cc.Node;
    component: cc.RenderComponent;

    belongDrawCallId: number;
    vertexData: Float32Array;
    indexData: Uint16Array;
}

// 真实DrawCall信息
export interface IDrawCallInfo {
    drawCallId: number;
    orderInFrame: number;
    primitiveType: number;

    optimizeSuggestion: string;

```

#### 3.2.2 快照管理策略
1.  内存限制：最多缓存最近10帧的完整快照，超过阈值自动释放最早的帧
2.  懒加载解析：顶点数据、像素数据仅在用户选中对应DrawCall时解析，避免每帧全量处理
3.  自动释放：帧快照切换时，自动释放上一帧的离屏缓冲与像素数据，避免内存泄漏

### 3.3 渲染回放控制层：核心交互能力实现
实现对标Unity Frame Debugger的暂停、单步回放、逐DrawCall预览能力，核心基于离屏渲染实现，不影响主画布。

#### 3.3.1 核心实现逻辑
1.  **渲染暂停**：通过重写`cc.Director.mainLoop`，暂停状态下跳过游戏场景的更新与渲染，仅保留调试器UI的更新，冻结当前帧的渲染数据。
2.  **离屏渲染缓冲创建**：基于2.4.x gfx层创建跨平台兼容的离屏帧缓冲，用于单步DrawCall回放。
```typescript
private _offscreenFrameBuffer: cc.gfx.FrameBuffer;
private _offscreenTexture: cc.Texture2D;

// 创建离屏渲染缓冲
private _createOffscreenBuffer() {
    const device = cc.gfx.Device.getInstance();
    const size = cc.view.getFrameSize();
    
    // 创建离屏纹理
    this._offscreenTexture = new cc.Texture2D();
    this._offscreenTexture.initWithSize(size.width, size.height);
    
    // 创建帧缓冲
    this._offscreenFrameBuffer = device.createFrameBuffer({
        width: size.width,
        height: size.height,
        renderTargets: [this._offscreenTexture.getGFXTexture()],
        depthStencilBuffer: device.getDepthStencilBuffer()
    });
}
```
3.  **单步回放执行**：基于当前帧快照的DrawCall列表，按顺序在离屏缓冲中逐个执行DrawCall，每执行一个就读取缓冲内容作为渲染结果。
```typescript
// 单步执行到指定DrawCall
public stepToDrawCall(drawCallIndex: number) {
    if (!this._currentSnapshot || !this._offscreenFrameBuffer) return;
    
    const device = cc.gfx.Device.getInstance();
    const gl = device._gl;
    
    // 绑定离屏帧缓冲，清空画布
    device.setFrameBuffer(this._offscreenFrameBuffer);
    device.clear(cc.gfx.CLEAR_COLOR | cc.gfx.CLEAR_DEPTH, cc.color(0, 0, 0, 0), 1, 0);
    
    // 按顺序执行到目标DrawCall
    const targetDrawCalls = this._currentSnapshot.drawCalls.slice(0, drawCallIndex + 1);
    for (const drawCall of targetDrawCalls) {
        this._executeDrawCall(drawCall);
    }
    
    // 读取渲染结果，展示到UI
    this._readOffscreenResult(drawCallIndex);
    
    // 恢复主画布帧缓冲
    device.setFrameBuffer(device.getDefaultFrameBuffer());
}

// 执行单个DrawCall
private _executeDrawCall(drawCall: IDrawCallInfo) {
    const device = cc.gfx.Device.getInstance();
    // 绑定纹理、材质、Shader
    device.setTexture(0, drawCall.texture.getGFXTexture());
    device.setUniforms(drawCall.material.getEffect(), drawCall.material.getProperties());
    device.setBlendFunc(drawCall.blendFunc.src, drawCall.blendFunc.dst);
    // 提交绘制
    device.draw(cc.gfx.PT_TRIANGLES, 0, drawCall.indexCount);
}
```
4.  **预览结果读取**：兼容WebGL与原生平台的像素读取，Web端直接转Canvas预览，原生端读取像素转Base64展示。

### 3.4 合批分析引擎：精准定位合批失败根因
基于2.4.x原生合批规则，实现全自动合批中断分析，是本方案的核心价值点。

#### 3.4.1 分析核心逻辑
```typescript
// 分析当前帧的合批中断点
private _analyzeBatchBreaks(snapshot: FrameSnapshot): IBatchBreakInfo[] {
    const breakInfos: IBatchBreakInfo[] = [];
    const drawCalls = snapshot.drawCalls;
    let breakIdCounter = 0;

    // 遍历相邻DrawCall，定位中断原因
    for (let i = 1; i < drawCalls.length; i++) {
        const prevDrawCall = drawCalls[i - 1];
        const currDrawCall = drawCalls[i];
        const breakCommand = currDrawCall.sourceCommands[0];
        let breakReason = '';
        let optimizeSuggestion = '';

        // 按优先级排查中断原因（完全匹配引擎合批规则）
        if (prevDrawCall.material !== currDrawCall.material) {
            breakReason = '材质实例不一致，相邻节点使用了不同的材质实例';
            optimizeSuggestion = '将需要合批的节点统一使用同一个材质实例，避免创建重复材质';
        } else if (prevDrawCall.texture !== currDrawCall.texture) {
            breakReason = '绑定纹理不一致，相邻节点使用了不同的图集/单图';
            optimizeSuggestion = '将相邻节点的精灵合并到同一张图集，确保使用同一纹理引用';
        } else if (prevDrawCall.blendFunc.src !== currDrawCall.blendFunc.src || prevDrawCall.blendFunc.dst !== currDrawCall.blendFunc.dst) {
            breakReason = '混合模式blendFunc不一致';
            optimizeSuggestion = '统一相邻节点的混合模式，避免频繁切换src/dst参数';
        } else if (breakCommand.globalZOrder !== prevDrawCall.sourceCommands[0].globalZOrder) {
            breakReason = 'globalZOrder不一致，导致渲染队列排序后不连续';
            optimizeSuggestion = '将需要合批的节点设置相同的globalZOrder，确保排序后连续';
        } else if (breakCommand.type !== 'QuadCommand') {
            breakReason = `插入了不可合批的${breakCommand.type}命令，强制flush合批`;
            optimizeSuggestion = '将Mask/Graphics/CustomCommand等不可合批节点集中放置，避免打断连续合批';
        } else if (prevDrawCall.vertexCount >= 4096) {
            breakReason = '合批顶点数超过引擎默认上限4096，强制拆分DrawCall';
            optimizeSuggestion = '拆分连续的Quad数量，或修改引擎MAX_VERTEX_BUFFER_SIZE配置提升上限';
        } else {
            breakReason = '渲染状态被修改（如Uniform值更新），强制flush合批';
            optimizeSuggestion = '避免在相邻节点渲染之间动态修改材质的Uniform值';
        }

        breakInfos.push({
            breakId: breakIdCounter++,
            drawCallBefore: prevDrawCall.drawCallId,
            drawCallAfter: currDrawCall.drawCallId,
            breakCommand,
            breakReason,
            optimizeSuggestion
        });
    }

    // 计算合批率
    const totalQuads = snapshot.totalQuads;
    const batchedQuads = drawCalls.reduce((sum, dc) => sum + dc.quadCount, 0) - drawCalls.length;
    snapshot.batchRate = totalQuads > 0 ? (batchedQuads / totalQuads) * 100 : 100;

    return breakInfos;
}
```

#### 3.4.2 特殊场景适配
1.  **Mask组件适配**：Mask会插入模板测试命令，强制打断合批，分析时会标记Mask的影响范围，建议将同Mask内的节点统一合批
2.  **Label合批适配**：系统字体Label无法合批（每个Label单独纹理），分析时会明确提示，建议替换为BMFont字体
3.  **动态材质修改适配**：检测到运行时修改材质Uniform的行为，会标记为合批中断原因，建议使用实例化材质或顶点色实现动态效果

### 3.6 平台适配层：WebGL与原生兼容处理
| 差异点 | WebGL平台适配 | 原生JSB平台适配 |
|--------|---------------|------------------|
| 离屏渲染 | 支持HTMLCanvasElement与FrameBuffer | 仅支持gfx.FrameBuffer，禁止使用HTMLCanvas |
| 像素读取 | canvas.toDataURL()直接获取 | 通过gl.readPixels读取像素数组，转Base64展示 |
| 纹理访问 | 可直接获取image对象 | 仅能通过gfx纹理对象读取像素数据 |
| 内存管理 | 浏览器自动GC，无特殊限制 | JSB内存管理严格，帧快照像素数据需手动释放，避免内存泄漏 |
| GL上下文 | 单上下文，无需频繁切换 | 原生上下文绑定严格，离屏渲染前后必须恢复默认帧缓冲 |

## 四、性能优化与安全规范
### 4.1 性能优化策略（调试器自身不影响游戏性能）
1.  **按需开启**：仅调试器开启时注入钩子，关闭时100%恢复引擎方法，零性能损耗
2.  **懒加载解析**：仅用户选中DrawCall时，才解析顶点数据、读取像素数据，避免每帧全量处理
3.  **刷新频率控制**：调试面板收起时，降低捕获频率，仅统计DrawCall数量，不解析详情
4.  **内存限制**：最多缓存10帧快照，超出自动释放旧数据，像素数据仅保留当前选中项
5.  **钩子精简**：钩子函数内仅做数据浅拷贝，复杂解析与分析放到帧结束后的空闲时间执行

### 4.2 安全与兼容规范
1.  **调试器自身渲染排除**：调试器UI节点设置独立group，钩子函数自动跳过，避免干扰调试数据
2.  **正式包一键剔除**：通过CC_DEBUG宏包裹所有调试代码，发布正式包时自动剔除，无残留
3.  **无侵入设计**：绝不修改引擎源码，所有功能通过原型注入实现，不影响引擎原生逻辑
4.  **异常兜底**：钩子函数内添加try/catch，即使调试器异常，也不会导致游戏渲染崩溃
5.  **原生平台兼容**：所有渲染API均使用2.4.x gfx层封装，禁止直接调用平台原生GL接口

## 五、扩展能力补充
基于本方案，可扩展以下进阶调试能力，进一步提升实用性：
